Apa itu Git?

Git adalah sistem versi kontrol yang paling umum digunakan saat ini dan dengan cepat menjadi standar untuk versi kontrol. Git bersistem kontrol versi terdistribusi, yang berarti salinan kode lokal kita terkontrol secara penuh di repositori. Repositori lokal yang berfungsi penuh ini memudahkan untuk bekerja secara offline atau dari jarak jauh. Anda mengikat pekerjaan Anda secara lokal, lalu menyinkronkan salinan repositori Anda dengan salinan di server. Paradigma ini berbeda dengan kontrol versi terpusat di mana klien harus menyinkronkan kode dengan server sebelum membuat kode versi baru.

Fleksibilitas dan popularitas Git menjadikannya pilihan yang tepat untuk tim mana pun. Banyak developer dan lulusan perguruan tinggi sudah mengetahui cara menggunakan Git. Komunitas pengguna Git telah menciptakan banyak sumber daya untuk melatih pengembang dan popularitas Git memudahkan Anda untuk mendapatkan bantuan saat Anda membutuhkannya. Hampir setiap lingkungan pengembangan memiliki dukungan Git dan alat baris perintah Git berjalan di setiap sistem operasi utama.

Dasar - dasar Git

Setiap kali Anda menyimpan pekerjaan kita, kita akan membuat commit di Git. Commit adalah snapshot dari semua file Anda pada satu titik waktu. Jika sebuah file tidak berubah dari satu commit ke commit berikutnya, Git menggunakan file yang disimpan sebelumnya. Desain ini berbeda dari sistem lain yang menyimpan versi awal file dan menyimpan catatan delta dari waktu ke waktu.

Commit membuat tautan ke Commit lain, membentuk grafik riwayat development. Kita dapat mengembalikan kode ke commit sebelumnya, memeriksa bagaimana file berubah dari satu Commit ke commit berikutnya, dan meninjau informasi seperti di mana dan kapan perubahan dibuat. commit diidentifikasi di Git oleh hash kriptografi unik dari konten commit. Karena semuanya berciri - ciri, tidak mungkin membuat adanya perubahan, kehilangan informasi, atau merusak file tanpa Git mendeteksinya.

Branches dalam Git

Setiap Developer menyimpan perubahan repositori kode lokalnya sendiri. Karena itu kita dapat memiliki banyak perubahan berbeda berdasarkan commit yang sama. Git menyediakan alat untuk mengisolasi perubahan dan kemudian menggabungkannya kembali. Branches, yang merupakan petunjuk ringan untuk pekerjaan yang sedang berjalan dan mengelola pemisahan repositori pada git. Setelah pekerjaan kita dibuat di branch selesai, kita bisa menggabungkan kembali ke cabang utama (atau master) pada repositori.

File dan Commits

File di Git memiliki salah satu dari tiga status: modified, staged, or committed. Saat kita pertama kali mengubah file, perubahan hanya ada di direktori kerja kita. Mereka belum menjadi bagian dari committ atau riwayat development. Kita harus mengatur file yang ingin diubah sertakan dalam commit (Kita dapat menghilangkan file yang ingin Anda commit secara terpisah). Area Stagging berisi semua perubahan yang akan kita sertakan dalam commit berikutnya. Setelah kita puas dengan file tersebut, lakukan dengan pesan diskripsi yang menjelaskan apa yang berubah. Commit ini menjadi bagian dari riwayat development.

Staging memungkinkan kita memilih perubahan file mana yang akan disimpan dalam commit sehingga kita dapat memecah perubahan besar menjadi serangkaian commit yang lebih kecil. Saat kita mengurangi ruang lingkup commit, akan lebih mudah untuk meninjau riwayat commit untuk menemukan perubahan file tertentu.
